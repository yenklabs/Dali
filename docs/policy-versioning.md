# Policy Versioning: Dali Reproducibility & Attribution Benchmark

The benchmark records a composite `policy_version` on every `CitationIntegrityResult`. This document defines what each sub-version covers, when to bump, and how cross-version comparison works.

## Why this exists

The benchmark's longitudinal value depends on stable scoring semantics. Without explicit policy versioning, later changes to the taxonomy, rubric, or scoring logic would silently invalidate earlier results, making "did integrity improve over time?" impossible to answer.

The composite version captures every dimension that, if changed, would make two results structurally incomparable.

Given the same corpus input and identical sub-version set, the benchmark is expected to produce semantically equivalent evaluation outputs.

## Composite version structure

```
taxonomy=<a>;rubric=<b>;scoring=<c>;normalization=<d>;schema=<e>
```

The value is generated by `corpus/policy.py` from five sub-version constants.

## Sub-versions

| Sub-version | Covers | Bump when |
|---|---|---|
| `taxonomy` | `CitationFailureClass` and `MutationType` enums | Adding, removing, or renaming a class or mutation type |
| `rubric` | `defensibility_risk` thresholds and risk classification semantics | Adjusting what counts as critical / high / medium / low |
| `scoring` | Existence, support, and integrity scoring logic | Changing scoring formulas, thresholds, or fallback chains |
| `normalization` | URL canonicalization, case-name normalization, citation-text canonicalization | Changing how raw inputs are normalized before comparison |
| `schema` | `CitationFailureCase` and `CitationIntegrityResult` shape | Adding, removing, or retyping a field |

Sub-versions follow semver: `MAJOR.MINOR.PATCH`.
- `MAJOR`: backwards-incompatible (renaming, removing, semantic change)
- `MINOR`: backwards-compatible addition
- `PATCH`: clarification or documentation change with no behavioral effect

## Cross-version aggregation

`runners/run_integrity.py` refuses to combine results from different policy versions into one aggregate. To opt in, pass `--allow-cross-version`. When the flag is set, the output JSON records `cross_version_aggregation: true` and lists every distinct `policy_version` present.

This is intentional friction: silent cross-version comparison is the most common way longitudinal benchmarks become misleading.

## Current version (v0.2.1)

```
taxonomy=2.0.0;rubric=1.0.0;scoring=1.0.0;normalization=1.0.0;schema=1.0.0
```

Sub-version numbers are independent from the overall benchmark release version. `taxonomy=2.0.0` refers to the second major revision of the failure-class vocabulary, not to Dali v2.
